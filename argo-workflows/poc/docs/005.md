# XXX-005: Job Dispatch Pattern - Multi-Cluster Execution

## Context

A core use case for Argo Workflows is end-to-end testing and operational procedures across systems. Many target systems are air-gapped, have no API, or have no dedicated test interface/service. The current manual approach is ad-hoc `kubectl exec` or port-forwarding to run trigger processes by direct database queries or hitting an endpoint if available.

The current dev environment consists of multiple clusters, and as Argo Workflows does not natively support multi-cluster orchestration, we need a pattern that enables remote job execution across clusters while maintaining security isolation and RBAC controls.

### Requirements

1. **Multi-cluster orchestration**: Execute jobs across dev-merit and dev-gcp clusters
2. **Namespace targeting**: Teams operate in separate namespaces with different RBAC policies
3. **Centralized control**: Single Argo installation to manage all workflows
4. **Cross-cluster dependencies**: Workflows need to pass data between steps that will be spawned in our primary cluster and managed namespace, but can execute tasks across namespaces either in the same or other cluster
5. **Security isolation**: Credentials remain in target namespaces, not passed through workflows
6. **RBAC granularity**: Each task runs with only the permissions it needs if needed

### Cluster Architecture

```
Cluster A (Primary - dev-merit)
  sys-argo-workflows namespace
    ├── argo-server              # UI and API
    ├── workflow-controller      # Manages workflows in sys-argo-workflows
    ├── workflow-runner SA       # ServiceAccount for workflow pods
    └── cluster-tokens Secret    # Contains tokens for remote clusters

Cluster B (Secondary - dev-others)
  any namespace (e.g., team-a, team-b)
    ├── workflow-remote-executor SA  # ServiceAccount for remote job execution
    ├── Jobs (created remotely)      # Actual workload
    └── namespace-specific SAs       # Optional: per-namespace permissions set on the job
```

## Decision

We adopt a **Job Dispatch Pattern** with cross-cluster support, where workflow pods create Kubernetes Jobs in target namespaces/clusters with custom ServiceAccounts (if needed) and resource limits.

### Pattern Overview

```
Argo Workflow
  → workflow pod (sys-argo-workflows namespace)
    → kubectl create Job (target namespace/cluster)
      → Job pod (with optional namespace-specific SA)
        → connects to DB/service via local DNS
        → runs script, exits
    → workflow pod retrieves logs or results via ARGO_RESULT_OUTPUT marker
    → workflow pod deletes Job
```

### Why Jobs over Pods or `pods/exec`

Jobs provide RBAC separation, we can grant `jobs.create` without `pods.create` or `pods/exec`, limiting the blast radius of workflow permissions. While both Jobs and Pods support ServiceAccounts and resource isolation equally, Jobs gives us:

- **Completion semantics**: `kubectl wait --for=condition=complete` vs pod phase checks
- **Automatic cleanup**: `ttlSecondsAfterFinished` handles garbage collection
- **Conceptual clarity**: Workflow pods create Jobs (no pod-ception), delegating execution to the Job controller

This introduces minimal overhead (Job resource + Pod) but provides operational convenience and tighter permission boundaries.

Considering the following point made in the [#infra-on-steriods channel](https://utilitywarehouse.slack.com/archives/C5DS918LQ/p1771564475143259?thread_ts=1771510015.743249&cid=C5DS918LQ)

> We ran Apache Airflow in the past and it was a mess, 100s of containers coming and going. This leads to an explosion in cardinality in cadvisor metrics. We had to write a bunch of Prometheus drop rules to keep our monitoring costs sane. There are second order effects like metrics that we expose for log throttling, these explode in cardinality too as they include container label.

I think we will need to figure out how to creately label and ignore. Also limit parallelism and its overall use.

## Implementation

### Authentication Flow

**Token Generation (one-time setup):**

1. Run [token generation script](../scripts/get-cluster-tokens.sh)
2. Script generates token FROM dev-gcp for `workflow-remote-executor` SA
3. Token is stored in the primary cluster as `cluster-tokens` Secret
4. Token represents: `system:serviceaccount:default:workflow-remote-executor`

**Workflow Execution (runtime):**

1. Workflow pod reads token from `cluster-tokens` Secret
2. Builds kubeconfig with token + CA certificate
3. kubectl commands authenticate AS `workflow-remote-executor`
4. Remote cluster grants permissions via ClusterRole/ClusterRoleBinding

### RBAC Model

**Primary Cluster (dev-merit):**

- `workflow-runner` SA: Orchestrates workflows, creates jobs locally ([example](../cluster/primary/workflow/sa.yaml))
- Namespace-scoped Roles for local job management
- Read access to `cluster-tokens` Secret

**Secondary Cluster (dev-merit/gcp):**

- `workflow-remote-executor` SA: Identity for token-based remote access ([example](../cluster/cluster-b/rbac.yaml))
- ClusterRole: Permissions to create/manage jobs cluster-wide
- Per-namespace SAs: Jobs run with namespace-specific permissions (optional, team-owned)

**Separation of Concerns:**

| Identity                 | Cluster   | Purpose               | Permissions                         |
| ------------------------ | --------- | --------------------- | ----------------------------------- |
| workflow-runner          | Primary   | Orchestrate workflows | Create jobs locally, read tokens    |
| workflow-remote-executor | Secondary | Remote job creation   | Create jobs anywhere (via token)    |
| default (or custom)      | Secondary | Execute job workload  | Access namespace secrets, resources |

### Result Passing Convention

Jobs output results using a standardized marker:

```bash
# Job execution
RESULT=$(jq -n '{status: "success", data: {...}}')

# Output (base64-encoded for multi-line support)
echo "ARGO_RESULT_OUTPUT=$(echo "$RESULT" | base64 -w0)"
```

Workflow extraction:

```bash
kubectl logs job/$JOB_NAME | sed -n 's/^ARGO_RESULT_OUTPUT=//p' > /tmp/result.txt
```

### Security Considerations

**Credentials stay in target namespaces:**

```yaml
# Job spec (created by workflow)
spec:
  template:
    spec:
      serviceAccountName: db-query-runner # Namespace-specific SA if needed
      containers:
        - name: task
          image: postgres:15
          env:
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef: # Secret in target namespace
                  name: db-credentials
                  key: password
```

**No credentials pass through workflow parameters or logs.**

## Consequences

- :white_check_mark: **Centralized orchestration**: Single Argo installation manages multi-cluster workflows
- :white_check_mark: **Security isolation**: Credentials remain in target namespaces, Jobs run with scoped ServiceAccounts
- :white_check_mark: **Standard Kubernetes primitives**: Uses Jobs (not custom controllers or CRDs)
- :white_check_mark: **Team autonomy**: Each namespace controls what it exposes via RBAC
- :x: **Non-native approach**: Argo doesn't support multi-cluster natively. This is a workaround
- :x: **Token management**: Requires periodic rotation via manual script or CI/CD automation
- :x: **Metrics cardinality**: Short-lived Jobs create metric explosion (similar to previous Airflow issues)
- :x: **No community support**: Standard Argo documentation assumes single-cluster operation

### Required

- Removing `pods/exec` from Argo controller/server RBAC (already done - not needed for operation)
- Each target namespace controls what it exposes via ServiceAccount/Role/RoleBinding
- Job cleanup on workflow failure must be handled via Argo's `onExit` handler or `ttlSecondsAfterFinished`
- Monitoring cost management: Prometheus drop rules for short-lived job metrics

## Caveats

This cross-cluster approach is a **tactical workaround**, not a natively supported feature of Argo Workflows. While the operational footprint remains confined to the primary cluster's namespace, the non-native nature introduces risks that warrant ongoing evaluation.

### Technical Limitations

:warning: **Non-native implementation**: Built on kubectl and token-based authentication rather than designed-for-purpose controls  
:warning: **Result passing fragility**: Depends on log parsing conventions and base64 encoding  
:warning: **Size constraints**: Results limited by Argo parameter storage (~256KB recommended), though we can use artifacts
:warning: **Token management**: Requires periodic rotation via manual script or CI/CD automation  
:warning: **No community support**: Standard Argo documentation assumes single-cluster operation

### What are the Alternatives

#### Community-suggested approaches

The free ones which all require additional infrastructure

- [Admiralty](https://admiralty.io/blog/2019/01/17/running-argo-workflows-across-multiple-kubernetes-clusters/) - multi-cluster scheduler
- [KubeStellar](https://dettori.medium.com/harnessing-the-power-of-multi-cluster-argo-workflows-with-kubestellar-9dd5582e76bb) - multi-cluster control plane
- [Open Cluster Management](https://github.com/sarabala1979/argo-wf-ocm-multicluster) - cluster federation
- Commercial tools: Pipekit, Karmada, etc. essentially cluster set/group management tooling

**Alternative workflow engines:**

- **Argo Workflows + Argo Events**: Requires event bus (NATS/Kafka) and Argo per cluster, introduces async complexity and operational overhead
- **Temporal**: Requires dedicated infrastructure, code-first paradigm (vs YAML), activities still call Kubernetes APIs for containerized tasks. Open-source but commercially driven with ambiguous free tier limitations for self-hosted production

> [!WARNING]  
> Token rotation strategy mirrors [feat deployment auth approach](https://github.com/utilitywarehouse/js-mono/blob/main/Makefile#L77). These are long-lived tokens - rotation mechanism or alternative auth needed.

> [!WARNING]  
> Jobs can use kubectl-capable containers. If a Job has RBAC to read secrets, those secrets will be captured in logs. This needs refinement. Potential options include admission controllers to block/enforce rules on allowed container images or commands. However, kubectl is required for our job dispatch pattern, creating a tension between capability and control.
