# XXX-006: Workflow Governance and Execution Control

## Context

Argo Workflows gives teams a powerful execution primitive: submit a `Workflow`, get arbitrary code running in the cluster. Without deliberate controls, anyone with submit access can define any image, any command, any script, and target any namespace their service account can reach.

As adoption grows across teams, the risk compounds, not just from malicious use, but from accidental misuse and from a lack of visibility into what is running, where, and on whose authority.

This write-up is to address governance across four layers:

- **Kubernetes layer** — what the workflow pod is permitted to do at the API level
- **Argo layer** — what Argo itself will and will not accept at admission
- **Repository layer** — how execution logic is structured, owned, and reviewed before it reaches the cluster
- **Identity and audit layer** — who submitted what, and how that is traceable

SSO is already committed to in [XXX-003](./003.md). The identity and audit trail that SSO provides is essential to a governance model, it means every workflow submission is attributable to someone.

## Goals

- Know what is running in the cluster, in which namespace, using which image, submitted by whom
- Prevent unreviewed execution logic from reaching the cluster
- Preserve flexibility for teams to compose workflows without pre-defining every combination in advance
- Make cross-team template reuse explicit and attributable, even where automated enforcement is not yet possible
- Provide layered controls so that no single bypass is catastrophic

## Decision

Governance is applied at four layers. Each layer is described below with what was decided and what was explicitly rejected.

### Layer 1: Kubernetes

#### Cluster/Namespace

Access is to be restricted to the platform team. ArgoCD should be the only applier for workflow and template resources and this repository the source of truth.

Direct kubectl apply is strongly discouraged but possible for admins, which is precisely why admin access must be a controlled and accountable group.

#### Workflow Pod Service Account

The service account used by workflow pods (`workflow-runner`) is granted the minimum permissions required to implement the ephemeral Job pattern [XXX-005](./005.md). The role permits creating, watching, and deleting Jobs, and reading pod logs at a cluster level because it needs to be cross-namespace. Nothing else.

This is enforced regardless of what the workflow defines. Even if a workflow contains logic it should not, the pod cannot act beyond this boundary at the Kubernetes API level.

#### Job Level Service Account

A Job runs can run with its own namespace-scoped service account. Any permissions a Job requires i.e. secret access, resource writes, external calls, belong here, not on workflow-runner. Keeping these concerns separate ensures that the risk radius of the workflow pod itself remains fixed and minimal regardless of what individual Jobs need to do.

> [!NOTE]
>
> Currently viewed as optional for teams who have put secrets under elevated restrictions beyond namespace. To be confirmed after testing on real UW clusters

### Layer 2: Argo — Controls

#### Auth

The Argo Server will intentionally not be exposed beyond its namespace. This will close off CLI submission as a path, ensuring that the subsequent layers cannot be bypassed. `Workflow`/`WorkflowTemplate` changes must go through a pull request and pass review before they are applied to the cluster.

Workflows are submitted/ran exclusively through the Argo UI, which requires SSO authentication and provides a full audit trail.

#### Template Validation

Argo has limited controls in restricting what can be specified in a Workflow. See their documentation on [Workflow Restrictions](https://argo-workflows.readthedocs.io/en/latest/workflow-restrictions/#available-restrictions) for more.

The configuration `templateReferencing: Secure` mode enforces that every workflow be submitted via `workflowTemplateRef`, meaning all orchestration logic must be pre-defined as a `WorkflowTemplate`.

We found this to be pretty useless and would prime us for unecessary bloat

- Every new combination of steps would require a new `WorkflowTemplate`, creating maintenance overhead
- Ad-hoc chaining of results between steps becomes impossible without pre-defining the chain

What we actually want is step-level enforcement `steps` and `dag` coordinator templates may be defined inline in a `Workflow` because they contain no execution logic, only wiring. Templates with actual execution logic — `container` and `script` — must always reference a `WorkflowTemplate` via `templateRef`.

Argo does not natively support this granularity. As a result, the Argo layer on its own cannot enforce this rule. Enforcement is therefore delegated to the repository layer (see below), with convention as a reinforcing mechanism.

### Layer 3: Repository

This is going to have to be the primary enforcement layer. It covers structure, ownership, review, version control, and deployment; more than any other layer due to positioning or lack of capability.

The following monorepo structure and CODEOWNERS configuration is designed so that execution logic cannot reach argo workflows for execution on any cluster without being reviewed by the team responsible for it.

#### Directory Structure

```
...
/ argo-workflows
├── cluster/                    # install configuration owned by platform teams
├── namespaced/                 # install configuration owned by platform teams
│    └── controller/
│    └── plugins/
│    └── server/
├── workflow-templates/
│   ├── _shared/                # Cross-team utility templates, owned by platform teams
│   │   └── CODEOWNERS
│   ├── payments/               # Templates owned and maintained by the payments team
│   │   └── CODEOWNERS
│   ├── data/                   # Templates owned and maintained by the data team
│   │   └── CODEOWNERS
│   └── team-<name>/            # One directory per team, per namespace
│       └── CODEOWNERS
│
├── workflows/                  # Flat owned by platform teams for now
│   └── CODEOWNERS
│   └── workflow-a.yaml
│   └── workflow-b.yaml
│   └── ...
└── ...
```

- `workflow-templates/` contains `WorkflowTemplate` resources — reusable, reviewed execution logic. These are the unit of governance. Changes here require review from the owning team's CODEOWNERS.
- `workflows/` contains `Workflow` resources — orchestration only. These wire together `WorkflowTemplate` steps. Platform temas will need to own until we can figure out how to codify enforcement. There also high level questions on ownership seeing as workflows can comprise of steps interacting with multiple systems across clusters/namespaces, we dont have a way to attribute a DRI or team for value streams yet.

#### CODEOWNERS Design

Each team directory under `workflow-templates/` has its own `CODEOWNERS` file assigning that team as reviewer. The platform team owns `_shared/` and `workflows/` for now

```
# workflow-templates/payments/CODEOWNERS
*   @org/team-payments

# workflow-templates/_shared/CODEOWNERS
*   @org/platform
# workflows/CODEOWNERS
*   @org/platform
```

This means:

- A PR adding or changing a `WorkflowTemplate` under `team-payments/` requires approval from `@org/team-payments`
- A PR adding or changing any `Workflow` will require approval from `@org/platform`
- A PR adding or changing any install configuration will require approval from `@org/platform`

This will `@org/platform` to remain informed of use cases we may get as we go. `@org/platform` includes

- @utilitywarehouse/quality-engineering
- ...

#### Review Requirements for WorkflowTemplates

Before a `WorkflowTemplate` is merged and applied to the cluster, the PR must include:

- A description of what the template does and why it exists
- The image(s) used, including registry source and tag pinning rationale
- The permissions the template requires (if any beyond the workflow-runner SA)
- Other access it requires to perform the operation i.e. cerbos, aws, gcp, git, etc.

Reviewers are responsible for checking the image source, the command being run, whether the template structure is appropriate, and whether the execution logic is consistent with what the description claims.

Merging will trigger a `kubectl apply` via Argo CD. A `WorkflowTemplate` is not available in the cluster until it has been merged and applied.

#### :warning: The Cross-Team Template Problem

The repository structure above enforces that changes to a team's `WorkflowTemplate`s require that team's approval. It does not enforce that a `Workflow` which references a template in `workflow-templates/team-Y/` requires `team-Y`'s approval.

This is a genuine gap and is why we've made the workflows directory a flat structure and owned by `@org/platform` until we find a better way.

### Layer 4: Identity and Audit

SSO means every workflow submission/run will be tied to a real user identity. Argo records the submitting identity in workflow metadata and in its own audit log. This provides post-hoc traceability: for any workflow that ran, we can determine who submitted it, when, and what templates it referenced.

This does not prevent a bad submission, but it ensures that every submission is attributable. Combined with the RBAC controls at Layer 1 (which bound what any submission can actually do), the risk radius of any given submission is contained even before the audit trail is consulted.

The audit log records submissions and workflow metadata, but does not surface template execution details at a glance. Determining exactly what ran in a namespace will require drilling into individual workflow step records which is accessible from the UI but is an intentional investigative step

## Other considerations

- **Cross-team approval enforcement via CI lint**
  The current model relies on convention for cross-team template usage. A CI lint step that parses `templateRef` references and derives required reviewers from a template ownership registry would close this gap. This is the recommended next step and should be implemented once the directory structure and CODEOWNERS design have stabilised across the first set of teams onboarded.
- **Admission webhook for inline execution logic**
  Argo does not natively prevent inline `container`/`script` templates in `Workflow` objects. A validating admission webhook (e.g. via OPA/Gatekeeper or Kyverno) could reject such submissions at the API server before they reach Argo. This would harden the convention into a technical control.
- **Template parameter governance**
  Argo's parameter validation is limited to which we currently exploit for our Job dispatch approach as we pass whole Job specs through it. Template design is the primary control. The ideal state is one where every `WorkflowTemplate` is purpose-built with bounded inputs, and reviewers are explicitly checking for parameter misuse as part of the review process; and that every `Workflow` uses these templates rather than inline scripting.
